## Название

UPD 19.04.2026:
В общем, сначала репозиторий назывался TsarGame как игра для Георгия Палыча, но я подумал, что нужно вывести на отдельный уровень идею, чем "сделать и забыть" / "заучить, здать, забыть" и т.д. Поэтому спустя какое-то время, когда я определился с сюжетом и +- тематикой я решил объединить два слова: Dream (Мечта, сон, грёзы) + Glen (узкая, уединённая долина, часто с протекающим через неё ручьём или рекой) и вывестив всё в DreamGlen, но, когда определился с географией игры, решил перейти к более лаконичной комбинации слов, выбрал слово не Glen, а isle. ## Сюжет

UPD 19.04.2026:
В N океане есть архипелаг островов (Много островов = закос на расширения игры). (Государства в игре != государстве на планете Земля, но планета ~ Земляподобной группы). Один из островов, ранее считавшийся мега-супер-пупер плодородным ныне почти заброшен, на нём живут единицы. Остров с фермой и плодородными полями будет полностью заброшен и будет считаться непригодным для жизни, нерентальбельным и тд, если не игрок не сделает иначе. Игрок - авантюрный чел, который решил взяться за поручения/миссию/контракт: "Сделай ферму на острове рентабельной, чтобы остров не закрылся, на всё-провсё даётся M годов, иначе Bad End, противном случае игрок станет владельцем фермы и будет крутым и тд". Сам остров (пока в моей голове) разделён на 5 частей, добавлю мб больше.
1. Южная часть - пристань, с пляжем, NPC рыбак + места для рабылка и всякое такое.
2. Центральная часть - Ферма с плодородными полями + мб место для строительства всяких построек-приблуд для фермерства (загоны, пруды, силосы, сараи, мб какие-то магические штуки)
3. Западная часть - волшебный лес с домиком NPC Ведуньи, генерации подземелий и тд. Добывается дерево, всякие семена, уникальные лесные мобы, коренья, грибы и прочее-прочее. Потом поподробнее расширю мысли. Аналог пещер.
4. Восточная часть - Гора-вулкан из разных уровней пещерной системы (можно вверх и вниз идти), рядом перед входом есть NPC шахтёр. Добывается камень, руды, ценные камни, уникальные пещерные мобы, ископаемые и тд. Аналог леса.
5. Северная часть - Небольшая деревня, которая раньше была небольшим городком, многие уехали. Есть NPC старейшина, продавщица, семья-рукодельников мастера и мастерицы, мб ещё кого-нибудь приплету.

## Алгоритмы (Предполагаемые)


1. **Фермерство / животноводство** — `FSM (конечный автомат) [ультра-легко]` для стадий роста культур и состояний животных (`голоден`, `сыт`, `спит`, `даёт ресурс`); `Weighted random + binary search [легко]` для качества урожая, редких семян и бонусного дропа; `Spatial hashing / Uniform Grid [средне]` для поиска ближайших предметов, животных и интерактивных объектов на ферме.

2. **Пивоварение / виноделие** — `FSM [ультра-легко]` для состояний перерабатывающих станков (`пусто`, `загружено`, `брожение`, `готово`); `Observer / Event pattern [ультра-легко]` для уведомлений о завершении переработки; `Weighted random + binary search [легко]` для определения качества напитка в зависимости от сырья и удачи.

3. **Рыбалка** — `Weighted random + binary search [легко]` для таблиц улова по зоне, сезону и времени суток; `PRD (pseudo-random distribution) [средне]` для более “честного” выпадения редкой рыбы без слишком длинных серий неудач; `FSM [ультра-легко]` для состояний мини-игры рыбалки (`ожидание`, `клёв`, `подсечка`, `выловлено`).

4. **Шахтёрное дело** — `BSP dungeon generation [сложно]` для процедурной генерации уровней шахты с комнатами и коридорами; `Flood fill / BFS [легко]` для проверки, что уровень проходим и все важные зоны достижимы; `A* [легко]` для перемещения врагов и NPC-шахтёра по тайловой карте; `Lightmap + multiply blending [сложно]` для темноты, факелов и освещения в пещерах.

5. **Собирательство** — `Weighted random + binary search [легко]` для выпадения ресурсов, семян, грибов, кореньев и редких находок; `Spatial hashing / Uniform Grid [средне]` для размещения и быстрого поиска ближайших собираемых объектов; `Perlin noise [сложно, опционально]` для более естественного распределения лесных ресурсов, полян и зон редких растений.

6. **Репутация** — `Observer / Event pattern [ультра-легко]` для начисления или снятия репутации при продаже товаров, выполнении просьб и общении с NPC; `FSM [ультра-легко]` или пороговая система состояний для уровней отношения (`недоверие`, `нейтралитет`, `дружелюбие`, `уважение`).

7. **Механика подсчёта очков прогресса** — `Observer / Event pattern [ультра-легко]` для сбора событий прогресса из разных систем игры; `Weighted scoring model` на основе таблицы весов для подсчёта вклада игрока в развитие острова; при необходимости `Circular buffer [легко]` можно использовать для отладочной визуализации темпа прогресса по дням.

8. **Показатель удачи** — `PRD [средне]` для контроля вероятностей редких событий; `Weighted random + binary search [легко]` для перерасчёта шансов дропа, рыбы, урожая и добычи; `Shuffle bag [ультра-легко, опционально]` для защиты от слишком длинных серий одинаковых результатов.

9. **Снаряжение** — `Spatial hashing / Uniform Grid [средне]` для реализации магнитизма предметов и поиска добычи в радиусе; `FSM [ультра-легко]` для состояний инструментов и оружия (`готов`, `используется`, `кулдаун`); `Observer / Event pattern [ультра-легко]` для автоматического пересчёта характеристик игрока после смены экипировки.

10. **HP + выносливость** — `Game loop + deltaTime [легко]` для корректной траты и регенерации параметров независимо от FPS; `FSM [ультра-легко]` для состояний персонажа (`обычный`, `устал`, `ранен`, `без сознания`); `AABB / circle overlap [ультра-легко]` для урона, подбора еды и взаимодействия с лечащими объектами.

11. **Смена дня / ночи** — `Lightmap + multiply blending [сложно]` для освещения мира, ночной затемнённости и локальных источников света; `Fixed timestep accumulator [легко]` для стабильного течения игрового времени; `Observer / Event pattern [ультра-легко]` для уведомления систем о наступлении утра, вечера и полуночи.

12. **Смена сезонов** — `FSM [ультра-легко]` для календаря (`весна`, `лето`, `осень`, `зима`); `Weighted random + binary search [легко]` для сезонных погодных событий, доступных культур и рыбы; `Observer / Event pattern [ультра-легко]` для обновления ассортимента магазинов, расписаний NPC и визуального оформления мира.

13. **Распорядок дня у NPC** — `Behavior tree tick traversal [средне]` для выбора текущего поведения NPC в зависимости от времени суток, погоды и состояния мира; `A* [легко]` для перемещения между домом, работой, магазином и другими точками; `FSM [ультра-легко]` можно использовать для простых NPC с минимальным набором состояний.

14. **Небольшая социальная система** — `FSM [ультра-легко]` для этапов отношений и состояний диалогов; `Observer / Event pattern [ультра-легко]` для реакции NPC на подарки, помощь, выполнение квестов и репутацию; `Weighted random + binary search [легко, опционально]` для выбора бытовых реплик и случайных диалоговых ответов.

15. **Побочные квестовые линии** — `FSM [ультра-легко]` для состояний квеста (`не начат`, `активен`, `условие выполнено`, `завершён`, `провален`); `Observer / Event pattern [ультра-легко]` для отслеживания выполнения условий без жёсткой связности между системами; `A* [легко, косвенно]` может использоваться, если квестовые NPC должны перемещаться по миру по сценарию.

16. **Механика роста и расширения острова в зависимости от проданных товаров** — `Observer / Event pattern [ультра-легко]` для фиксации всех продаж и вкладов игрока; `Weighted scoring / threshold system` для перехода острова между этапами развития (`упадок`, `восстановление`, `рост`, `процветание`); `FSM [ультра-легко]` для состояний развития ключевых зон и открытия новых возможностей.

17. **Базовые настройки игры (сохранения, настройки, внутренняя консоль)** — `Observer / Event pattern [ультра-легко]` для применения настроек без жёсткой связи систем; `Circular buffer (FPS counter) [легко]` для отладки и встроенного отображения производительности; `Command + undo/redo [легко, опционально]` можно использовать для внутренней debug-консоли и тестовых инструментов разработчика.

## Используемые структуры данных

Для проекта `Dreamisle` планируется использовать как базовые, так и более специализированные структуры данных. Их выбор связан не с формальностью, а с конкретными задачами игры: хранением мира, поиском пути, генерацией уровней, работой инвентаря, NPC и интерактивных объектов.

### 1. Матрица / двумерный массив (`list[list[Tile]]`)
Основная структура для представления тайлового мира.
Использование:
- хранение карты острова;
- хранение процедурно сгенерированных уровней шахты и магического леса;
- проверка проходимости тайлов;
- поиск соседних клеток;
- определение типов поверхности (`земля`, `вода`, `камень`, `грядка`, `трава`).

Почему нужна:
- тайловый мир естественно представляется в виде сетки;
- удобно использовать для `BFS`, `A*`, проверки коллизий и генерации.

### 2. Граф поверх тайловой карты
Логически карта представляется как граф: тайл = вершина, переход между соседними тайлами = ребро.
Использование:
- поиск пути для NPC и мобов;
- проверка достижимости нужных зон;
- анализ связности процедурно созданного уровня.

Почему нужна:
- алгоритмы `A*` и `BFS` работают именно с графами;
- даже если отдельный граф явно не хранится, карта всё равно интерпретируется как граф.

### 3. Очередь (`collections.deque`)
Использование:
- `BFS` и `Flood Fill`;
- обработка волновых алгоритмов;
- возможные игровые очереди событий и действий.

Почему нужна:
- эффективное добавление и удаление элементов с концов;
- стандартная структура для обхода в ширину.

### 4. Приоритетная очередь / куча (`heapq`)
Использование:
- алгоритм `A*` для поиска пути;
- возможное управление приоритетом задач NPC;
- обработка событий по важности или стоимости.

Почему нужна:
- позволяет быстро извлекать следующий наиболее приоритетный узел;
- обязательна для эффективной реализации `A*`.

### 5. Хеш-таблица / словарь (`dict`)
Использование:
- хранение данных предметов по `id`;
- хранение конфигурации NPC;
- таблицы лута;
- быстрый доступ к сохранённым параметрам;
- индексация сущностей, ресурсов, квестов, расписаний.

Почему нужна:
- в Python это одна из самых важных и быстрых структур;
- даёт быстрый доступ к данным по ключу.

### 6. Множество (`set`)
Использование:
- хранение посещённых тайлов в `BFS` и `A*`;
- список уже собранных наград;
- активные флаги квестов;
- проверка уникальности объектов и состояний.

Почему нужна:
- быстрые операции проверки принадлежности;
- удобно для логики прогресса и обходов графа.

### 7. Пространственный хеш / Uniform Grid
Специализированная пространственная структура данных для игрового мира.
Использование:
- быстрый поиск ближайших предметов;
- реализация магнитизма лута;
- поиск объектов рядом с игроком;
- оптимизация взаимодействия с животными, ресурсами, врагами и NPC.

Почему нужна:
- наивная проверка “каждый объект с каждым” даёт плохую производительность;
- разбивка мира на ячейки позволяет проверять только близкие объекты.

### 8. Дерево BSP (Binary Space Partitioning)
Использование:
- процедурная генерация шахты;
- рекурсивное деление уровня на области;
- размещение комнат и соединение их коридорами.

Почему нужна:
- `BSP` естественно строится как дерево;
- удобно хранить родительские и дочерние области генерации.

### 9. Таблица префиксных сумм / накопленных весов
Использование:
- выбор предметов из таблиц лута;
- определение качества рыбы, урожая, руды;
- обработка вероятностей с учётом удачи игрока.

Почему нужна:
- используется вместе с `weighted random + binary search`;
- позволяет быстро выбирать результат по весам.

### 10. Список (`list`)
Использование:
- инвентарь игрока;
- содержимое сундуков;
- список NPC;
- список активных квестов;
- список объектов на сцене;
- пути, найденные алгоритмом.

Почему нужна:
- базовая структура для упорядоченных коллекций;
- удобна для отрисовки, обновления и сериализации.

### 11. Стек
Использование:
- обход некоторых деревьев;
- возможная отладочная история действий;
- отдельные механики генерации и обработки состояний.

Почему нужна:
- полезен в некоторых алгоритмах и системах редакторского/отладочного характера;
- в Python может быть реализован обычным `list`.

### 12. Кольцевой буфер (Circular Buffer)
Использование:
- хранение последних значений FPS;
- отладочная статистика производительности;
- анализ стабильности игрового цикла.

Почему нужна:
- позволяет хранить ограниченное окно последних измерений без бесконечного роста памяти.

### 13. Таблицы расписаний NPC
Практически это структура вида:
`dict[season][day_type][time_range] -> action/location`
Использование:
- распорядок дня NPC;
- маршруты по времени суток;
- сезонные изменения поведения жителей.

Почему нужна:
- расписание удобнее хранить как данные, а не как длинные `if-else`;
- это упрощает расширение количества NPC.

### 14. Таблицы состояний / конечные автоматы
Практически это набор состояний и переходов между ними.
Использование:
- рост культур;
- поведение станков;
- состояния NPC;
- квесты;
- календарь и сезоны.

Почему нужна:
- позволяет явно и формально описывать логику игровых систем;
- уменьшает хаос в коде.

### Наиболее важные структуры данных для проекта

Если выделять минимальный набор самых важных структур данных для `Dreamisle`, то это:
- матрица тайлового мира;
- граф карты;
- очередь (`deque`);
- приоритетная очередь (`heapq`);
- словарь (`dict`);
- множество (`set`);
- пространственный хеш / `Uniform Grid`;
- дерево `BSP`;
- таблицы весов для случайного выбора.

Именно они напрямую поддерживают ключевые механики проекта: карту, генерацию уровней, поиск пути, лут, NPC и оптимизацию взаимодействий в игровом мире.
